위키독스 책은 한 가지 방식으로만 작성해야 하는 것은 아닙니다.  
저자의 작업 방식, 사용하는 도구, 원고를 관리하는 습관에 따라 여러 방법으로 책을 만들 수 있습니다.

이 점은 위키독스의 큰 장점입니다.  
브라우저에서 바로 쓰고 싶으면 웹 편집기를 사용하면 되고, 평소 쓰는 편집기나 자동화 도구가 있다면 그 흐름을 위키독스와 연결할 수 있습니다.

다만 앞으로 이어지는 설명의 중심은 `깃허브 연동 방식`입니다.  
다른 작성 방식은 위키독스가 제공하는 선택지로 간단히 소개하고, 자세한 내용은 참고 URL을 통해 살펴보는 정도로만 다루겠습니다.

## 위키독스에서 직접 작성하기

가장 단순한 방법은 위키독스 웹 사이트에서 바로 글을 작성하는 것입니다.  
별도의 도구를 설치하지 않아도 되고, 브라우저만 있으면 책의 페이지를 만들고 내용을 입력할 수 있습니다.

처음 위키독스에 책을 만들어 보는 저자라면 이 방식만으로도 충분히 책을 시작할 수 있습니다.  
웹 편집기 중심의 작성 방법이 궁금하다면 아래 문서를 참고하면 됩니다.

> 참고: [위키독스 책 만들기](https://wikidocs.net/196756)

## 옵시디언 플러그인으로 작성하기

옵시디언을 사용한다면 위키독스 책을 내려받아 로컬에서 작성하고 관리할 수 있습니다.  
마크다운 파일을 기준으로 원고를 다루기 때문에 평소 옵시디언으로 메모와 문서를 정리하는 사람에게 잘 맞습니다.

옵시디언 플러그인 방식은 위키독스와 옵시디언을 직접 연결해 사용하는 흐름입니다.  
플러그인 사용 자체를 중심으로 책을 관리하고 싶다면 아래 문서를 참고하면 됩니다.

> 참고: [위키독스 옵시디언 플러그인](https://wikidocs.net/267526)

## MCP를 사용하여 작성하기

MCP를 사용하면 도구나 LLM이 위키독스 책과 직접 상호작용하면서 원고를 작성하고 수정할 수 있습니다.  
예를 들어 페이지 목록을 확인하고, 특정 페이지의 내용을 읽고, 새 페이지를 만들거나 기존 페이지를 고치는 흐름을 자동화할 수 있습니다.

이 방식은 위키독스와 외부 도구를 더 직접적으로 연결하고 싶을 때 살펴볼 만합니다.  
MCP를 활용한 자동화 방식이 궁금하다면 아래 문서를 참고하면 됩니다.

> 참고: [위키독스 MCP](https://wikidocs.net/289752)

## 깃허브와 연동하여 작성하기

가장 중요하게 살펴볼 방식은 원고를 깃허브 저장소에서 관리하고 위키독스와 연동하는 방법입니다.  
책의 각 페이지를 마크다운 파일로 작성하고, 목차 파일을 통해 책의 구조를 관리하는 방식입니다.

깃허브를 사용하면 수정 이력이 자연스럽게 남습니다.  
어떤 문장을 언제 바꾸었는지 확인할 수 있고, 필요하면 이전 상태로 돌아가거나 변경 내용을 비교할 수 있습니다.

공동 집필에도 유리합니다.  
여러 사람이 함께 원고를 고치거나, pull request를 통해 수정 제안을 받고 검토한 뒤 반영하는 흐름을 만들 수 있습니다.

여러 도구와 함께 쓰기에도 좋습니다.  
로컬 편집기에서 원고를 작성하고, LLM에게 초안을 만들게 하거나 문장을 다듬게 하고, 변경된 내용을 깃허브에 기록한 뒤 위키독스에 반영하는 흐름을 만들 수 있습니다.

책을 꾸준히 고쳐 나갈수록 이런 이력 관리의 장점은 더 커집니다.  
원고를 하나의 프로젝트처럼 다루고 싶다면 깃허브 연동 방식이 특히 잘 맞습니다.

## 깃허브 연동 방식으로 진행하기

이후 장에서는 깃허브 연동 방식만을 기준으로 실제 작업 흐름을 이어 갑니다.  
위키독스에서 깃허브 연동 책을 만들고, 자동 생성된 저장소를 살펴본 뒤, 그 저장소를 로컬로 내려받아 옵시디언에서 편집하는 흐름입니다.

원고 편집 과정에서는 옵시디언을 작업 공간으로 사용하고, 옵시디언 안의 터미널에서 Codex나 Claude Code 같은 LLM 도구를 함께 활용합니다.  
수정한 내용은 깃허브에 push하고, 위키독스에서 반영 결과를 확인합니다.

구체적인 준비물과 실제 작업 순서는 다음 장부터 하나씩 살펴보겠습니다.  
목표는 복잡한 개발 지식을 익히는 것이 아니라, 내가 쓰고 싶은 내용을 안정적으로 책으로 만들어 가는 것입니다.
