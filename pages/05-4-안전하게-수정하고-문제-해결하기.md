MCP는 편리하지만, 책 내용을 직접 바꿀 수 있는 강력한 방식입니다.  
따라서 작업 전후를 확인하고, 수정 범위를 작게 나누어 진행하는 습관이 필요합니다.

## 수정 전 확인하기

페이지를 수정하기 전에는 먼저 현재 내용을 읽어 보는 것이 좋습니다.  
LLM에게 곧바로 수정하라고 요청하기보다, 먼저 페이지 내용을 확인하고 수정 제안을 받아 봅니다.

```text
페이지 ID 456의 내용을 읽고, 수정이 필요한 부분을 먼저 목록으로 정리해 주세요.
아직 위키독스에는 반영하지 마세요.
```

제안을 확인한 뒤 실제 반영을 요청합니다.

```text
방금 제안한 수정 중 1번과 3번만 페이지 ID 456에 반영해 주세요.
제목과 공개 여부는 바꾸지 마세요.
```

이렇게 하면 의도하지 않은 대규모 수정이나 구조 변경을 줄일 수 있습니다.

## update_page는 필요한 항목만 바꾸기

페이지 수정에는 `update_page` 도구가 사용됩니다.  
이 도구는 제목, 내용, 부모 페이지, 공개 여부 중 필요한 항목만 바꾸는 방식으로 동작합니다.

예를 들어 제목만 바꿀 때는 제목만 전달하면 됩니다.

```json
{
  "page_id": 456,
  "subject": "새 제목"
}
```

내용만 바꿀 때는 본문만 전달합니다.

```json
{
  "page_id": 456,
  "content": "새로운 내용"
}
```

공개 여부를 바꿀 때는 `open_yn` 값을 사용합니다.

```json
{
  "page_id": 456,
  "open_yn": "N"
}
```

부모 페이지를 제거해 최상위 페이지로 이동하려면 `parent_id`에 `-1`을 사용합니다.

```json
{
  "page_id": 456,
  "parent_id": -1
}
```

LLM에게 요청할 때도 이 원칙을 그대로 적용합니다.  
“제목만 바꿔 주세요”, “본문만 수정해 주세요”, “공개 여부만 바꿔 주세요”처럼 수정 범위를 분명히 말합니다.

## 큰 책을 다룰 때의 순서

페이지가 많은 책에서는 전체 내용을 한 번에 읽는 방식이 비효율적입니다.  
다음 순서로 작업하는 것이 좋습니다.

1. `list_books`로 책 목록을 확인합니다.
2. `get_book_toc`로 목차와 페이지 구조를 확인합니다.
3. 수정할 페이지를 정합니다.
4. `get_page`로 해당 페이지 본문만 읽습니다.
5. 수정안을 먼저 확인합니다.
6. 필요한 부분만 `update_page`로 반영합니다.

이 방식은 LLM이 불필요한 내용을 처리하지 않게 해 주고, 저자도 수정 범위를 쉽게 확인할 수 있습니다.

## 연결이 실패할 때

MCP 연결 상태가 실패로 표시되면 먼저 토큰과 서버 주소를 확인합니다.

- 서버 URL이 `https://wikidocs.net/mcp`인지 확인합니다.
- 인증 헤더가 `Authorization: Token 발급받은토큰` 형식인지 확인합니다.
- 토큰 앞뒤에 불필요한 공백이나 따옴표가 들어가지 않았는지 확인합니다.
- 설정을 바꾼 뒤 LLM 도구를 다시 실행했는지 확인합니다.

터미널에서 직접 확인해야 한다면 다음과 같은 방식으로 도구 목록 호출을 테스트할 수 있습니다.

```bash
curl -X POST https://wikidocs.net/mcp \
  -H "Content-Type: application/json" \
  -H "Authorization: Token 발급받은토큰" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}'
```

도구 목록이 출력되면 토큰과 서버 연결은 정상입니다.  
그래도 LLM 도구에서 실패한다면 해당 도구의 MCP 등록 방식이나 설정 파일 위치를 다시 확인합니다.

## 권한 오류가 날 때

권한 오류가 발생하면 접근하려는 책이나 페이지가 내 계정에 연결되어 있는지 확인합니다.  
MCP는 내가 저자로 등록된 책과 페이지에 대해서만 작업할 수 있습니다.

먼저 책 목록을 다시 확인합니다.

```text
내 위키독스 책 목록을 보여 주세요.
```

수정하려는 책이 목록에 없다면 해당 계정으로는 그 책을 관리할 수 없습니다.  
위키독스에서 로그인 계정이 맞는지, 책의 저자 권한이 있는지 확인해야 합니다.

## 수정 결과가 기대와 다를 때

수정 결과가 기대와 다르다면 바로 다시 큰 수정을 요청하지 말고, 먼저 현재 페이지 내용을 다시 읽습니다.

```text
페이지 ID 456의 현재 내용을 다시 보여 주세요.
```

그다음 잘못된 부분만 좁혀서 요청합니다.

```text
페이지 ID 456에서 두 번째 문단만 이전 표현에 가깝게 다시 다듬어 주세요.
다른 문단은 수정하지 마세요.
```

위키독스에는 페이지 수정 이력이 남습니다.  
잘못 반영된 내용이 있다면 위키독스 페이지의 변경 이력을 확인해 이전 버전으로 되돌릴 수 있습니다.

## 토큰과 원고 보안

API 토큰은 원고에 절대 넣지 않습니다.  
예제 코드에도 실제 토큰을 쓰지 말고 `발급받은토큰` 같은 자리 표시자를 사용합니다.

LLM에게 여러 파일을 읽게 할 때는 민감한 정보가 포함되어 있지 않은지 확인합니다.  
계정 정보, 개인 연락처, 비공개 메모, 실제 토큰이 원고 폴더에 섞여 있으면 안 됩니다.
