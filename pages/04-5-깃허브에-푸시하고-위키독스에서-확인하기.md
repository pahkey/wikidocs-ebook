앞 장에서는 옵시디언 안에서 터미널을 열고 LLM 도구를 함께 사용하는 방법을 살펴보았습니다.  
이제 옵시디언에서 수정한 원고를 깃허브에 올리고, 그 결과가 위키독스 책에 반영되는 흐름을 확인해 보겠습니다.

옵시디언에서 파일을 저장하는 것만으로는 위키독스 책이 바뀌지 않습니다.  
로컬 컴퓨터에서 수정한 내용을 깃허브에 push해야 위키독스가 변경 내용을 가져와 책에 반영할 수 있습니다.

## 깃허브 푸시가 필요한 이유

옵시디언에서 원고를 수정하면 내 컴퓨터의 파일만 바뀝니다.  
예를 들어 `pages/` 폴더의 원고를 고치거나 `assets/` 폴더에 이미지를 추가해도, 그 상태는 아직 로컬 저장소 안에만 있습니다.

위키독스의 깃허브 연동 책은 깃허브 저장소의 내용을 기준으로 책을 업데이트합니다.  
따라서 로컬에서 수정한 내용을 커밋하고 깃허브에 push해야 위키독스가 새 원고를 확인할 수 있습니다.

흐름은 다음과 같습니다.

```text
옵시디언에서 원고 수정
-> git commit
-> git push
-> 위키독스 자동 반영
-> 위키독스 책 화면에서 확인
```

여기서 `commit`은 변경 내용을 하나의 기록으로 남기는 작업이고, `push`는 그 기록을 깃허브 저장소로 올리는 작업입니다.

## 터미널에서 직접 푸시하기

가장 기본적인 방법은 터미널에서 직접 git 명령을 실행하는 것입니다.  
앞 장에서 설정한 옵시디언 터미널을 사용해도 되고, 평소 사용하는 시스템 터미널을 사용해도 됩니다.

먼저 변경된 파일을 확인합니다.

```bash
git status
```

어떤 문장이 바뀌었는지 확인하려면 다음 명령을 실행합니다.

```bash
git diff
```

변경 내용에 문제가 없다면 책 원고와 이미지 파일을 스테이징합니다.

```bash
git add README.md TOC.md pages assets
```

그다음 커밋을 만듭니다.

```bash
git commit -m "원고 수정"
```

마지막으로 깃허브에 push합니다.

```bash
git push
```

## LLM에게 푸시를 맡기기

Codex나 Claude Code 같은 LLM 도구를 사용하고 있다면 git 작업을 도와 달라고 요청할 수도 있습니다.  
이 경우에도 변경 내용을 확인하는 책임은 저자에게 있습니다.

예를 들어 다음과 같이 요청할 수 있습니다.

```text
변경 내용을 확인하고, 원고와 이미지 파일만 스테이징한 뒤 커밋하고 push해 주세요.
커밋하기 전에 git diff 요약을 먼저 보여 주세요.
```

LLM이 명령을 실행하기 전에 어떤 파일을 스테이징하려는지 확인합니다.  
특히 `.obsidian/` 같은 개인 설정 폴더나 임시 파일이 포함되지 않았는지 살펴보는 것이 좋습니다.

LLM에게 맡기더라도 흐름은 직접 실행할 때와 같습니다.  
변경 내용을 확인하고, 커밋하고, 깃허브에 push한 뒤 위키독스에서 결과를 확인합니다.

## 옵시디언 플러그인으로 처리하기

터미널 명령이 익숙하지 않다면 옵시디언 플러그인을 활용할 수도 있습니다.  
`Git` 플러그인을 설치하면 옵시디언 패널 안에서 변경 내용을 확인하고, diff를 살펴보고, 커밋과 push까지 처리할 수 있습니다.

먼저 옵시디언 설정의 `커뮤니티 플러그인`에서 `Git`을 검색해 설치하고 활성화합니다.  
이 플러그인은 옵시디언 보관함을 git 저장소로 다룰 수 있게 해 주며, 변경 파일 확인, 스테이징, 커밋, push 같은 작업을 패널에서 처리할 수 있게 해 줍니다.

![](../assets/Pasted%20image%2020260427122552.png)

설치 후 명령어 팔레트에서 `Git: Open source control view`를 실행하면 소스 제어 패널을 열 수 있습니다.  
이 패널에서는 변경된 파일 목록을 확인하고, 파일별 diff를 열어 어떤 문장이 바뀌었는지 살펴볼 수 있습니다.

변경 내용을 확인한 뒤에는 필요한 파일만 스테이징하고, 커밋 메시지를 입력한 다음 push를 실행합니다.  
이렇게 하면 터미널에 git 명령을 직접 입력하지 않아도 옵시디언 안에서 기본적인 git 작업을 처리할 수 있습니다.

변경 내용을 확인하고, 커밋하고, push해야 깃허브와 위키독스에 반영됩니다.

## 깃허브에서 먼저 확인하기

push가 끝났다면 먼저 GitHub 저장소를 열어 변경 내용이 올라갔는지 확인합니다.  
저장소 첫 화면에서 최신 커밋 메시지와 시간이 방금 작업한 내용과 맞는지 확인합니다.

수정한 파일을 열어 원고가 제대로 올라갔는지도 확인합니다.  
이미지를 추가했다면 `assets/` 폴더에 이미지 파일이 올라갔는지, 원고의 이미지 링크가 올바른지도 함께 살펴봅니다.

![](../assets/Pasted%20image%2020260427123013.png)

## 위키독스에서 확인하기

깃허브에 push하면 위키독스가 변경 내용을 감지해 책에 반영합니다.  
반영까지 약간의 시간이 걸릴 수 있으므로 잠시 기다린 뒤 위키독스 책 화면을 새로고침합니다.

위키독스에서는 다음 내용을 확인합니다.

- 수정한 페이지의 문장이 제대로 보이는지
- 새로 추가한 이미지가 깨지지 않고 표시되는지
- `TOC.md`를 수정했다면 목차 순서와 하위 장 구조가 의도대로 보이는지
- `README.md`를 수정했다면 책 화면의 제목과 설명이 올바르게 보이는지

수정 내용이 보이지 않는다면 먼저 GitHub 저장소에 push가 제대로 되었는지 확인합니다.  
그다음 push한 브랜치가 위키독스와 연결된 브랜치인지, 파일 경로가 `TOC.md`와 정확히 맞는지 살펴봅니다.

![](../assets/Pasted%20image%2020260427123116.png)


## 이전 버전으로 되돌리기

깃허브를 사용하는 가장 큰 장점 중 하나는 이전 버전으로 돌아갈 수 있다는 점입니다.  
원고를 잘못 수정했거나 이미지 링크가 깨졌다면, 이전 커밋을 기준으로 되돌릴 수 있습니다.

터미널에서는 먼저 커밋 기록을 확인합니다.

```bash
git log --oneline
```

특정 커밋의 변경을 되돌리고 싶다면 `git revert`를 사용할 수 있습니다.

```bash
git revert 커밋ID
git push
```

`git revert`는 이전 기록을 지우지 않고, 되돌리는 내용을 새로운 커밋으로 남깁니다.  
책 원고처럼 기록이 중요한 작업에서는 기존 기록을 삭제하는 방식보다 이 방법이 더 안전합니다.

특정 파일만 이전 버전으로 되돌리고 싶다면 GitHub의 파일 히스토리나 옵시디언 Git 플러그인의 변경 기록을 확인한 뒤, 필요한 부분만 복원할 수도 있습니다.  
복원한 뒤에는 다시 커밋하고 push해야 위키독스 책에도 되돌린 내용이 반영됩니다.

[[TIP]]

초보 단계에서는 `git reset --hard`처럼 작업 내용을 강제로 되돌리는 명령은 조심하는 것이 좋습니다.  
잘못 사용하면 로컬에서 아직 커밋하지 않은 수정 내용이 사라질 수 있습니다.

[[/TIP]]

## 정리

옵시디언은 원고를 편집하는 공간이고, 깃허브는 그 원고를 기록하고 위키독스에 전달하는 통로입니다.  
원고를 수정한 뒤에는 변경 내용을 확인하고, 커밋하고, push한 다음 위키독스에서 실제 책 화면을 확인하는 습관을 들이면 좋습니다.
